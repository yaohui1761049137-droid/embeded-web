# CLAUDE.md — Embedded Web Project

## 项目概述

基于 Boa 0.94.13 的嵌入式 Web 服务器，包含 CGI 应用程序（登录/登出/动作处理）。

- **Web 服务器**: Boa 0.94.13（轻量级单线程 Web 服务器）
- **CGI 程序**: `login.cgi`、`logout.cgi`、`action.cgi`、`main.cgi`
- **构建系统**: Makefile
- **部署**: `deploy_ssh.py`（远程部署）、`package/deploy.sh`（本地打包部署）
- **目标平台**: 嵌入式 Linux 开发板

## 构建与部署

```bash
make          # 构建 boa 和 CGI
make deploy   # 部署到目标板
```

## AI 编程行为约束

以下规则来自 `.claude/AI_coding.md`，适用于所有 AI 辅助编码任务。

### 1. 先思考，再编码

**不要猜测。不要隐藏困惑。暴露权衡。**

实现之前：
- 明确说明你的假设。如果不确定，先问。
- 如果有多种解释，列出它们——不要默默选择。
- 如果存在更简单的方案，说出来。在必要时提出反对。
- 如果某件事不清楚，停下来。说出困惑点。提问。

### 2. 简洁优先

**用最少的代码解决问题。不要写推测性代码。**

- 不做要求之外的功能。
- 不为一次性代码做抽象。
- 不添加未经请求的"灵活性"或"可配置性"。
- 不为不可能的场景做错误处理。
- 如果你写了 200 行而它可以只有 50 行，重写它。

问自己："资深工程师会觉得这过于复杂吗？"如果是，简化。

### 3. 精确修改

**只动必须动的代码。只清理自己造成的垃圾。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使你会有不同做法。
- 如果发现无关的死代码，提一下——不要删除它。

当你的修改产生了遗留物：
- 删除 YOUR 修改导致不再使用的导入/变量/函数。
- 除非被要求，不要删除预先存在的死代码。

检验标准：每一行修改都应该直接追溯到用户的需求。

### 4. 目标驱动执行

**定义成功的标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 Bug" → "编写能复现它的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，给出简要计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

### 5. Sudo 密码处理

需要执行 `sudo` 命令时，先停下来找我输入密码，不要试图绕过 sudo 或寻找变通方案。
